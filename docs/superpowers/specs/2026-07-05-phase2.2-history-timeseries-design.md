# Phase 2.2 Design — Transaction History & Balance Time-Series

**Date:** 2026-07-05
**Status:** Draft — awaiting review
**Depends on:** Phase 1 (persistence) and Phase 2.1 (transactions + API). See `CLAUDE.md`,
`README.md`, and `docs/superpowers/specs/2026-07-04-phase2.1-transaction-operations-design.md`.

This designs **Phase 2.2 only**: the two read queries that power the Account
Overview page — a **paginated transaction history** (infinite scroll) and a
**running-balance time-series** (for the line chart). It excludes PDF export
(**Phase 2.3**) and the Angular SPA (**Phase 3**).

---

## 1. Scope

- **Paginated history**: the customer's transactions for one account, newest-first,
  loaded incrementally (infinite scroll).
- **Balance time-series**: one `{time, balance}` point per transaction, computed as a
  running balance via a SQL **window function** (the reason Spring Data JDBC was
  chosen — JPQL cannot express `SUM(...) OVER (...)`).
- Both live in the **`com.homework.history`** feature, are blocking Spring MVC, and
  reuse the header-based API conventions and money-as-string JSON from Phase 2.1.

**Out of scope:** PDF (2.3), Angular (Phase 3), and date-range chart filtering (the
`CLAUDE.md` §4 nice-to-have — deferred).

---

## 2. Endpoints (header-based)

Identifiers travel in headers (as in 2.1); pagination params are query string (they
are not identifiers). Ownership is enforced: `X-Account-Id` must belong to
`X-Customer-Id` → `404` otherwise.

| Method & path | Headers | Query | Result |
|---|---|---|---|
| `GET /api/account/transactions` | `X-Customer-Id`, `X-Account-Id` | `limit` (default 20, max 100), `cursor` (opaque, optional) | `HistoryPageResponse` |
| `GET /api/account/balance-series` | `X-Customer-Id`, `X-Account-Id` | — | `BalanceSeriesResponse` |

### DTOs (`DtoResponse` for full response bodies, `Dto` for nested components)
- `HistoryItemDto { transactionId, type, amount, currency, description, createdAt }`
  — nested component (`amount` a `BigDecimal` → JSON string; `createdAt` ISO-8601).
- `HistoryPageDtoResponse { items: HistoryItemDto[], nextCursor: String | null }`
  — full response (`nextCursor` is `null` on the last page).
- `BalancePointDto { time: Instant, balance: BigDecimal(→string) }` — nested component.
- `BalanceSeriesDtoResponse { points: BalancePointDto[] }` — full response.

## 3. Keyset (cursor) pagination — history

Chosen over offset because it is the correct infinite-scroll pattern: **stable when
new transactions are inserted** (no duplicated/skipped rows across pages) and
efficient at any depth.

- **Order:** `created_at DESC, transaction_id DESC` (newest first).
- **Cursor:** an **opaque** string encoding the last item's `(createdAt, transactionId)`
  — Base64 of `"<createdAt>|<transactionId>"`. The tiebreaker is `transaction_id`
  (not the technical row `id`): on a single account each operation writes at most one
  leg, so `transaction_id` is unique per row there — a valid keyset that keeps the
  technical PK out of the domain/DTO entirely. Encoded/decoded in the controller;
  the use case works with a typed cursor (`Instant createdAt, String transactionId`)
  or `null`.
- **Query** (parameters via `NamedParameterJdbcTemplate`); the keyset predicate is
  omitted on the first page:
  ```sql
  SELECT transaction_id, type, amount, currency, description, created_at
  FROM account_transaction
  WHERE account_id = :accountId
    AND (:cursorCreatedAt IS NULL
         OR created_at < :cursorCreatedAt
         OR (created_at = :cursorCreatedAt AND transaction_id < :cursorTxnId))
  ORDER BY created_at DESC, transaction_id DESC
  LIMIT :limit
  ```
- **`nextCursor`:** if exactly `limit` rows are returned, the cursor of the last row;
  otherwise `null`. (Simple and adequate; an extra "fetch limit+1 to detect more"
  refinement is unnecessary at demo scale.)

`account_id` here is the customer's **business `account_id`**; a customer account
only ever holds its own single-currency legs, so no ledger legs leak in.

## 4. Running-balance time-series — window function

```sql
SELECT created_at,
       SUM(CASE WHEN type = 'CREDIT' THEN amount ELSE -amount END)
           OVER (ORDER BY created_at ASC, id ASC) AS balance
FROM account_transaction
WHERE account_id = :accountId
ORDER BY created_at ASC, id ASC
```

- One point per transaction, **chronological** (time increasing on the x-axis).
- Signed amount: `CREDIT` adds, `DEBIT` subtracts (same rule as `LedgerWriter`).
- The final running-sum equals the account's current cached `balance` — a built-in
  consistency check.
- Returns the **whole** series (all points). Date-range filtering is deferred
  (nice-to-have).

## 5. Component design (four-package, in `com.homework.history`)

- **domain**
  - `BalancePoint` — value: `Instant time`, `BigDecimal balance`.
  - `HistoryItem` — value: `String transactionId`, `TransactionType type`,
    `BigDecimal amount`, `Currency currency`, `String description`, `Instant createdAt`.
  - `HistoryCursor` — value: `Instant createdAt`, `String transactionId` (typed cursor).
- **port**
  - `GetTransactionHistoryUseCase` — input port:
    `HistoryPage history(Long customerId, Long accountId, int limit, HistoryCursor cursor)`
    returning a `HistoryPage { List<HistoryItem> items, HistoryCursor nextCursor }`.
  - `GetBalanceSeriesUseCase` — input port:
    `List<BalancePoint> balanceSeries(Long customerId, Long accountId)`.
  - `AccountHistoryQuery` — output port for the read model:
    `List<HistoryItem> page(Long accountId, int limit, HistoryCursor cursor)` and
    `List<BalancePoint> balanceSeries(Long accountId)`.
- **usecase**
  - `GetTransactionHistoryService`, `GetBalanceSeriesService` — implement the input
    ports; each calls `AccountAccess.requireOwned(customerId, accountId)` (reused from
    `account.usecase`, the same cross-feature service the transaction use cases use)
    then delegates to `AccountHistoryQuery`.
- **infrastructure**
  - `infrastructure/controller`: `HistoryController` (both endpoints, header-based),
    DTOs, and cursor encode/decode.
  - `infrastructure/query`: `AccountHistoryJdbcQuery` implementing `AccountHistoryQuery`
    with `NamedParameterJdbcTemplate` (the two SQL queries above). This is a
    **CQRS-style read model** that reads `account_transaction` directly and therefore
    couples to that schema — an accepted trade-off on the query side, documented here.

## 6. Testing strategy (TDD, in-memory H2)

- **`AccountHistoryJdbcQuery` — balance series:** after a credit `+100` then debit
  `-30` on an account, the series is `[100.00, 70.00]` in chronological order.
- **`AccountHistoryJdbcQuery` — keyset paging:** insert N transactions; page with
  `limit` and a cursor; assert newest-first order, correct `nextCursor`, and that
  paging covers every row exactly once (no dup/skip).
- **`HistoryController` (`MockMvc`):** history endpoint returns items + `nextCursor`
  (money as strings); second call with the cursor returns the next window; an account
  not owned by `X-Customer-Id` → `404`; balance-series returns chronological points
  whose last value equals the account balance.

## 7. Definition of Done (Phase 2.2)

- `GET /api/account/transactions` returns a keyset-paginated, newest-first history
  with a working `nextCursor`; ownership enforced.
- `GET /api/account/balance-series` returns the chronological running-balance series
  via the window function; last point equals the current balance.
- Money as JSON strings; header-based; RFC-9457 errors reused.
- Queries live in the `history` feature; covered by tests.
- `./gradlew.bat clean build` is green.
